1.	Introduction 
This document shows how to use git lab and template to create a Pull Request correctly. 
 
2.	Git Lab Issue Board 
To control task easily, use the issues board in git lab to create new issue to log task. 
2.1	Create Labels 
Labels are using to mark a task need how much effort to do it. And tasks can have 1, 2, 3, 5, 8, 13 points. The higher point, the more effort you need to do a task. 
Go to Manage => Labels => New label => Insert number to Title Box => Create label. 
  
 
 
 
 
 
 
 
 
 
2.2	Create milestones 
Milestones will be a sprint. The sprint will have a start date and end date. 
Go to Plan => Milestones => New milestone => Insert name of sprint, start date, end date => Create milestone 
  
2.3	Create issue 
Issue is a user story that you need to complete. To finish an issue, you need to complete all sub tasks of it. Go to Plan => Issues => New issue => Insert information => Create issue 
Require field: 
-	Title: The name of the user story/issue. 
-	Assignee: Who has main responsibilities of user story/issue. - 	Labels: mark one of labels have created (1, 2, 3, 5, 8, 13) - 	Milestones: select the sprint. 
  
2.4	Create child tasks 
Select the issue that needs to break down the tasks. Then click Add and Insert Information then Create Task 
   
3.	Git Control 
Git is most important tool in software development. So, if you want to be a good developer, you must follow some of the git control rules. 
3.1	Git Branching Strategy 
The image below is a git branching strategy in development phase. Each branch has their own meaning. 
  
-	Main Branch (master/main): This branch only merge branches that have been thoroughly tested and accepted, usually merged with Release Branch. 
-	Develop Branch: This branch is used to merge with Feature Branches and Hotfix. 
-	Feature Branch: This branch is used for developing new features. Once the implementation is complete, this branch will be merged into Develop Branch. 
-	Hotfix Branch: This branch is used to fix critical bugs in the code. After local testing is completed, merge this branch into Main and Develop Branches.  
-	Release Branch: This branch is used to prepare the features for release and created from Develop Branch. Whenever have a release branch, must have a release note. 
 
3.2	Branching Naming Rule 
Besides the Main and Develop branch, the others have a naming rule when created new. 
3.2.1	Feature Branch Naming Rule 
When implementing new features, should create 2 branches for 2 points 
-	Implementation: features/Implementation_UserStoryName 
-	Design: features/Design_UserStoryName 
The implementation branch is used to implement code, and the design branch is used to create or update a design. 
Front-End Developer: The design branch needs to have Figma Design. 
Back-End Developer: The design branch needs to have API Design. 
3.2.2	Hot Fix Branch Naming Rule 
In software development, always have bugs and need to fix, then create branch to fix the bug and test at local before merge. The branch name should be hotfix/Bug_UserStoryName 
3.2.3	Release Branch Naming Rule 
For release the features are implemented and tested, create a branch from develop branch to release features and use that branch in Sprint Demo. The branch name should be release/sprint_x with x is number of sprints. Example: For release features for Sprint 1, then create release/sprint_1 
3.3	Commit 
3.3.1	Why Need Commit Quality 
Writing clear and structured commit messages helps: 
-	Understand the history of changes 
-	Facilitate code reviews and debugging 
-	Automate changelogs and release notes 
-	Improve team collaboration and traceability 
3.3.2	Rule 
-	Enough clear and easily to understand what change. 
-	Avoid commit “Update”, “Fix bug” but don’t know update or fix what. 
-	Focus in one change. 
-	Every commits should include only one point: Add Feature, Fix Bug, Refactor, etc. 
-	Avoid commit many changes in once but not related. 
 
4.	Pull Request 
For merge a new implementation feature for Sprint, always need a pull request to track and verify the implementation. The pull request must have below format, just copy and paste. Then need to rewrite the Change Description part based on Feature of the Pull Request. 
4.1	Implementation Format 
**Definition of Done** 
*	[ ] Feature works as expected and passes manual testing 
*	[ ] Code follows project conventions and is lint-free 
*	[ ] Unit and integration tests are written and passing 
*	[ ] No sensitive data or hardcoded secrets 
*	[ ] Feature is documented (README, API docs) 
*	[ ] CI/CD pipeline passes successfully 
*	[ ] Code is reviewed and approved 
--- 
**Review Checklist** 
*	[ ] PR is linked to a Jira/Trello task or issue 
*	[ ] PR title follows naming convention ([Feature], [Fix], [Refactor], etc.) 
*	[ ] Code is modular and follows SOLID principles 
*	[ ] Proper error handling and edge case coverage 
*	[ ] API responses are consistent and documented 
*	[ ] Git history is clean (no debug commits, proper messages) 
*	[ ] No unused code, commented-out blocks, or console logs 
*	[ ] Feature is integrated and does not break existing flows

4.2	Design Format 
**Definition of Done (DoD)** 
*	[ ] Design meets business and technical requirements 
*	[ ] All screens/components are covered 
*	[ ] Naming conventions and structure are consistent 
*	[ ] Design is reviewed and approved by stakeholders 
*	[ ] Figma file or documentation is shared and accessible 
*	[ ] Feedback from previous reviews has been addressed 
--- 
**Review Checklist** 
*	[ ] All design assets are linked or embedded 
*	[ ] Design follows brand guidelines 
*	[ ] UX flow is logical and intuitive 
*	[ ] API documentation includes request/response examples 
*	[ ] No missing states (loading, error, empty, etc.) 
*	[ ] Responsive behavior is considered (mobile/tablet/desktop)
